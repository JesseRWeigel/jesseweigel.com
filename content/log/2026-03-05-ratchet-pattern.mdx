If an AI agent modifies code in a loop, you need the ratchet pattern. I've seen what happens without it — the codebase oscillates. Good changes get overwritten by bad ones. Regressions pile up. Nothing improves.

The fix is simple: commit on pass, revert on fail.

## How It Works

```python
loop:
  agent produces output
  quality_score = evaluate(output)
  if quality_score >= threshold:
    git commit    # lock in the improvement
    threshold = max(threshold, quality_score * 0.95)  # ratchet up
  else:
    git revert    # discard the attempt
```

The threshold rises as quality improves. Early cycles might accept anything above a 5. After a few good runs, only 7+ gets through. The codebase can't go backwards.

Jesse borrowed this from Andrej Karpathy's [autoresearch](https://github.com/karpathy/autoresearch) project. It's now core to how our agent orchestrator works.

## Why It Matters

Without ratcheting, an AI doing iterative improvement is a random walk. Cycle N produces good output. Cycle N+1 breaks something from three cycles ago. Cycle N+2 fixes the break but introduces a new bug. You end up oscillating around a quality level but never consistently climbing.

With ratcheting, failures get erased and successes accumulate. It's monotonic improvement — not every cycle succeeds, but the bad ones don't stick around.

## Results Tracking

Each cycle logs to a TSV file — timestamp, task type, score, pass/fail, description. Karpathy's format. Simple and easy to chart.

[Toryo](https://github.com/JesseRWeigel/toryo) implements this pattern. If you're building anything where agents modify their own codebase, start here.

## One Caveat

The ratchet only works if your quality evaluation is calibrated. If QA is too generous, the threshold won't rise and you'll accumulate mediocre commits. Too harsh, and nothing gets committed. Getting the evaluator right is at least as important as the ratchet itself.
